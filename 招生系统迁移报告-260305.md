## 三、后端架构（Supabase → Express + SQLite）

### 3.1 新增 SQLite 数据表（backend/db.js）

| 表名 | 用途 | 主要字段 |
|------|------|---------|
| `screener_help_config` | 帮助信息（营业时间、联系方式、备注） | id (固定'default'), hours, contact, notes |
| `screener_snapshots` | 版本备份 | id, label, timestamp, data (JSON) |
| `screener_settings` | 通用键值存储（API 配置、权重、阈值等） | id, value (JSON), updated_at |
| `screener_candidates` | 人才库 | id, name, email, data (JSON), created_at, updated_at |

### 3.2 API 路由（backend/screenerRoutes.js）

所有路由挂载在 `/api/screener` 前缀下：

| 方法 | 路径 | 说明 |
|------|------|------|
| `GET` | `/help-config` | 获取帮助信息 |
| `PUT` | `/help-config` | 更新帮助信息 |
| `GET` | `/snapshots` | 获取所有快照 |
| `POST` | `/snapshots` | 创建快照 |
| `DELETE` | `/snapshots/:id` | 删除快照 |
| `GET` | `/settings/:id` | 获取单项配置 |
| `PUT` | `/settings/:id` | 保存/更新配置 |
| `POST` | `/settings/batch` | 批量获取配置 |
| `GET` | `/candidates` | 获取所有候选人 |
| `POST` | `/candidates` | 新增候选人 |
| `PUT` | `/candidates/:id` | 更新候选人 |
| `DELETE` | `/candidates/:id` | 删除候选人 |
| `POST` | `/candidates/bulk` | 批量导入候选人 |

| 步骤 | 命令/行为 | 说明 |
|------|----------|------|
| 1. 密钥配置 | `cp server_ssh_key ~/.ssh/business_server` | 将项目目录下的私钥复制到 SSH 标准位置 |
| 2. 连接测试 | `ssh root@8.145.43.200` | 验证网络和密钥可达性 |
| 3. 代码上传 | `rsync -avz` | 增量同步，排除 node_modules / .git / data.db / 密钥 |
| 4. 远程部署 | `deploy.sh` | 在服务器 ~/workspace/business 执行构建 + PM2 重启 |

- IP：`8.145.43.200`

## 八、注意事项

### ⚠️ .env.local 的 VITE_BACKEND_URL

当前设为 `http://localhost:3001`。如果生产环境前后端同机部署，无需修改；如果分离部署，需改为实际后端地址。

### 🔐 管理员密码为硬编码

`authService.ts` 中管理员账密直接写在代码里。如需更改，需修改源码并重新部署。建议后续迁移至数据库或环境变量。

### 🗄️ 生产数据库不会被覆盖

`upload-to-server.sh` 排除了 `backend/data.db`，所以每次部署不会覆盖生产数据。首次部署时数据库会自动创建空表。

### 🔌 部署前提

运行 `QUICK_DEPLOY.sh` 前需要确认：
1. 项目根目录存在 `server_ssh_key` 文件
2. 本机已安装 rsync 和 ssh 客户端
